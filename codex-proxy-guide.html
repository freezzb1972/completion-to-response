Codex Proxy 使用指南

多后端智能路由 · Responses API → Chat Completions · 2026-05-13

1. 架构概述

Codex Proxy 是一个多后端智能路由代理，由两层组成：

  • codex-gateway.js（Node.js 网关）— 接收 Codex CLI 请求，按模型名路由到不同后端
  • completion-to-response（Go 二进制）— 每个后端一个实例，将 Responses API 转换为 Chat Completions
Codex CLI ── :8082 ──▶ codex-gateway.js ── :8083 ──▶ converter (DeepSeek) ──▶ api.deepseek.com
Codex CLI ── :8082 ── codex-gateway.js ── :8084 ──▶ converter (NVIDIA) ──▶ integrate.api.nvidia.com

8. API Key 注入机制

Codex CLI 发送的是自己的 Key，不是后端 Key。网关的工作流程：

  1. Codex CLI → 网关（携带 Codex 自己的 Key）
  2. 网关查找 model_map，确定目标后端
  3. 网关替换 Authorization 头为后端 Key（从 codex-switch.toml 读取）
  4. 网关修改 body.model 为后端模型名
  5. 转发到对应后端的 converter 端口
结论：Codex CLI 中填任意 Key 即可，真正的后端 Key 存在 codex-switch.toml 中，不暴露给 Codex。

14. 修复记录

2026-05-13 — DeepSeek V4 思考模式兼容

修复 1：流式工具调用丢失

原因：DeepSeek 流式工具调用分两个阶段 — 第一个 chunk 含 id/type/name 但 arguments 为空，后续 chunk 只含 arguments 不含 type。转换器原有逻辑要求 type=="function" && arguments!=""，导致两个阶段都被跳过。

修复：添加 toolCallState 跨 chunk 累积状态，按 index 跟踪每个工具调用。首次遇到时记录 id/name，后续 chunk 累积 arguments 并发送 delta 事件。流式结束时创建 function_call 输出项。

修复 2：多轮对话空 role 错误

原因：buildMessages 函数未处理 input 数组中的 function_call（工具调用）和 function_call_output（工具结果）项，导致空 role 传给后端。

修复：添加 function_call → 助手消息（含 tool_calls）转换；添加 function_call_output → 工具角色消息转换。

修复 3：reasoning_content 传递

原因：DeepSeek V4 思考模式要求多轮对话中传递 reasoning_content，否则报错 "must be passed back to the API"。

修复：流式转换器累积 reasoning_content 并存入 function_call 输出项的 summary 字段。后续请求中从 summary 提取 reasoning_content 并传递给后端。
